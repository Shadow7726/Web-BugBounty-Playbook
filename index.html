Reconnaissance
Information gathering is the foundation of every successful bug bounty find. The more you know about your target, the more attack surface you expose. Never skip this phase.

Golden Rule of Recon
The bug is always in the part of the target you haven't looked at yet. Go wide before going deep. Most hunters lose bounties because they test what they can see ‚Äî not what they can find.

// Phase Six
Authentication & Session Attacks
Authentication failures are the second most common critical finding. JWT flaws, weak password resets, MFA bypass, and session fixation ‚Äî all lead to complete account takeover.
ACCOUNT TAKEOVER JWT ATTACKS MFA BYPASS
JWT Attack Playbook
  • 01
    Algorithm None ‚Äî Remove Signature
    Set alg: "none" in header, remove signature. Server accepts unsigned token if misconfigured. Use: jwt_tool TOKEN -X a
  • 02
    RS256 ‚Üí HS256 Algorithm Confusion
    Grab server's RSA public key from /jwks.json. Sign a forged token using the public key as HMAC-SHA256 secret. Server verifies thinking it's HS256. Use: jwt_tool TOKEN -X k -pk public.pem
  • 03
    Weak Secret Brute Force
    Many apps use weak HMAC secrets like "secret", "password", app name. Crack with: jwt_tool TOKEN -C -d rockyou.txt or hashcat -m 16500 token.jwt wordlist.txt
  • 04
    Kid Header SQL Injection
    The kid parameter tells server which key to use ‚Äî inject SQL: {"kid": "x' UNION SELECT 'attacker_secret'--"} then sign with that string. Server queries your injected key.
  • 05
    JWK Header Injection
    Embed your own public key in the jwk header. If the server uses the embedded key without validating its origin, you can forge any token. Use: jwt_tool TOKEN -X i
MFA Bypass Techniques
üîÑ
Response Manipulation
Intercept POST /verify-otp response. Change "success":false to "success":true. Many apps trust client-side response state.
‚è≠Ô∏è
Step Skip
After entering password, directly navigate to /dashboard without completing the OTP step. Does the server enforce step completion?
üîç
OTP in Response
Check if the OTP is returned in the API response body. Seen in poorly designed mobile apps: {"otp":"123456","message":"sent"}
üí®
Null / 000000 OTP
Submit empty, null, or 000000 as the OTP. Some apps skip validation for default/test values left in production.

// Phase Seven
Broken Access Control & IDOR
The #1 OWASP finding. IDOR, privilege escalation, forced browsing, and parameter manipulation. Found in 94% of applications. Extremely high impact, often easy to find.
IDOR PRIV ESCALATION FORCED BROWSE
IDOR Testing Methodology
  • 01
    Create Two Accounts ‚Äî Attacker & Victim
    Register two separate accounts. Log all requests made by both. Note every ID, UUID, or reference that appears in URLs, bodies, headers, and cookies.
  • 02
    Swap Object References Cross-Account
    Use attacker's session to request victim's objects: GET /api/profile/VICTIM_ID ‚Äî does the server return victim's data without error?
  • 03
    Test All HTTP Methods
    A GET endpoint might check authorization but a PUT or DELETE on the same path might not. Try all methods: GET, POST, PUT, PATCH, DELETE, HEAD, OPTIONS.
  • 04
    Check POST Bodies & JSON Fields
    IDOR isn't just in URLs. Look for user_id, account_id in POST bodies, hidden form fields, and JSON payloads.
  • 05
    Mass Assignment Attack
    Send extra fields the API doesn't advertise: {"name":"John","role":"admin","isAdmin":true,"credits":99999} ‚Äî does the server accept and apply the hidden fields?
Privilege Escalation Checklist
  • Access admin-only endpoints as a regular user: /admin/users, /api/admin/settings
  • Change your role in the request body: {"role": "admin"} in update profile
  • Add privilege headers to regular requests: X-User-Role: admin, X-Admin: true
  • Modify JWT payload claims: change "role":"user" to "role":"admin"
  • Access other org's data in multi-tenant apps by changing org_id or tenant_id
  • Test all endpoints as unauthenticated user ‚Äî some may not require tokens at all
  • Check if OPTIONS method bypasses access controls on restricted endpoints

Business Logic Flaws
Scanners can't find these. You must think like a developer, then like a criminal. Business logic bugs are application-specific, high-impact, and often lead to direct financial damage.
MANUAL ONLY HIGH IMPACT RACE CONDITIONS

The Scanner-Blind Category
No automated tool will ever find business logic bugs. You must deeply understand how the feature is supposed to work, then systematically ask: "What if I do this out of order? What if I send a negative value? What if I do this twice at exactly the same time?"

Elite Hunter Mindset
The difference between a $50 report and a $50,000 report is rarely skill ‚Äî it's mindset, persistence, and thinking beyond the obvious. This is what separates top hunters.
STRATEGY MINDSET ELITE
üéØ
Go Where Others Don't
The main app is the most tested. New features, mobile APIs, beta subdomains, and recently acquired products are goldmines. Watch for new asset additions in program scope.
üîó
Chain Vulnerabilities
A low-severity CORS issue + open redirect + XSS = critical account takeover chain. Always ask: how can I chain this with something else to make it worse?
üß™
Read Source Code
JS bundles, mobile APKs, and GitHub repos expose internal API endpoints, business logic, and hardcoded secrets that black-box testing never finds.
‚è∞
Be First on New Scopes
Set up alerts for program scope changes. New domains and new features have the lowest discovery rate. Speed and preparation beat skill in the first 48 hours after a scope addition.
üìö
Study Disclosed Reports
HackerOne disclosed reports are a masterclass. Study the methodology, not just the bug. Understand what the hunter thought, what they tried, and how they escalated impact.
ü§ù
Build Reputation
Quality over quantity. Programs invite top hunters to private programs. A 10-report year with five criticals beats 100 reports of low findings. Signal-to-noise ratio matters more than volume.
